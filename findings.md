## Findings

- `ConDiv_symlay/run_remote.sh` is intentionally hard-coded to a single 48-task Slurm allocation and then dispatches workers via `srun`, so the cluster only sees one job ID even though multiple worker steps run inside it.
- `ConDiv_symlay/ConDiv_mem.py` combines three concerns in `run_minibatch()`: staging the minibatch forcefield inputs, running each target worker, and aggregating divergences into the FF update. The new Slurm workflow needs those concerns split so simulation jobs and the update job can be separate Slurm jobs.
- The existing minibatch logic already tolerates worker failures as long as at least one target succeeds. The new dependency chain should therefore use `afterany` rather than `afterok` for the update job.
- The staged workflow must preserve `CONDIV_SYMLAY_LAYER_MANIFEST` during both initial round staging and update-job resubmission; otherwise the temporary membrane file can skip the symmetry projection step.
- User correction: the simulation side must be one Slurm array `sbatch` file per round, not one `sbatch` file per protein.
- The simplest compliant array design is one generated `simulate_array.sbatch` file per round plus the round manifest; `SLURM_ARRAY_TASK_ID` indexes into the manifest task order and resolves the per-target worker spec.
- Cluster evidence from `epoch_00_minibatch_00` shows the `upside` simulation itself finishes and writes `.run.*.h5` outputs, but the worker aborts before writing `<target>.divergence.pkl`; the dependent update job then fails with `All jobs failed`.
- In `ConDiv_symlay/ConDiv_mem.py`, the missing `*.divergence.pkl` can only happen after simulation completion and before the final pickle write, so the likely failure surface is the native-engine contrast path in `compute_divergence()` rather than the Slurm wrapper or the `upside` simulator subprocess.
- The first helper-subprocess mitigation fixed clean teardown for some targets, but the downloaded `test_dimer3` round showed a second failure mode: `3ukm` succeeds on 150-frame halves while the full 300-frame batch aborts, so some native crashes are batch-size dependent rather than pure teardown failures.
- The robust mitigation is two-layered: run contrast evaluation in a helper subprocess that skips `free_deriv_engine` on exit, and recursively bisect any failing frame batch until native evaluation succeeds on smaller chunks. Because `compute_divergence()` is frame-independent, concatenating chunk results is mathematically equivalent to the original full-batch result.
- Verification against the downloaded `ConDiv_symlay/test_dimer3/epoch_00_minibatch_00` round succeeded for all 15 targets with the new helper + recursive batch split path.
- The downloaded round exposed one more finalize-time mismatch: recovered contrast tensors can have empty inner blocks while the checkpoint parameter tensors keep nonzero inner shapes. Aligning empty gradient blocks to zero arrays matching the checkpoint parameter shapes makes the FF update and pair regularizer addition well-defined.
- With recovered `*.divergence.pkl` files present and the empty-block alignment in place, `finalize_minibatch_from_outputs()` succeeds on the downloaded `epoch_00_minibatch_00` data.
- User correction: `ConDiv_symlay/slurm-46899622.out` already came from the current code, so the helper retry path is still failing on the active cluster code path.
- On the local downloaded data, the current helper still crashes child Python processes under normal operation. A direct `2oar` replay showed a helper abort on a 44-frame chunk, and repeated stress-target probes showed that the crash behavior is nondeterministic rather than a single fixed-size threshold.
- Forcing the helper subprocess to single-threaded `OMP`/`BLAS` settings removes some failures (`2oar` and `5vb2` at batch 64) but does not eliminate them all (`3ukm` still aborts at larger batch sizes).
- Replacing the helper entrypoint with a minimal script that imports only `numpy`, `tables`, and `upside_engine`, plus switching `upside_engine.Upside.__init__` from `PyTables` metadata reads to `h5py`, reduces noise and exposed a real remaining single-frame native abort (`2oar` finite-equil frame index 173 in one replay still exited `-5`).
- Sampled single-frame helper evaluations can succeed repeatedly, so the remaining instability appears to be inside the native derivative path itself rather than only in Python teardown or large-batch concatenation.
- Root cause identified in native code: both `MembraneCBPotential::get_param_deriv()` and `MembraneSurfCBPotential::get_param_deriv()` in `src/membrane_potential.cpp` allocated a derivative vector of length `np` and then wrote right-side spline contributions at `shift + np`, causing out-of-bounds writes and corrupting the helper process during membrane-gradient evaluation.
- Fixing those two out-of-bounds writes removes the observed helper instability on the downloaded round:
  - `2oar`, `3ukm`, and `5vb2` all complete divergence recovery with zero helper retries under the normal helper path
  - a fresh local `finalize-round` replay on a clean temp copy of `test_dimer3` rebuilds all 15 divergence files with no `RECOVER_DIVERGENCE_FAIL` or `MISSING_DIVERGENCE`
- The native fix does not alter simulation trajectories: a direct old-vs-new `upside` binary comparison on the same seeded `2oar.run.1.h5` config produced exact matches for every shared `/output` dataset, including `pos`, `potential`, `kinetic`, and the membrane-relevant outputs.
